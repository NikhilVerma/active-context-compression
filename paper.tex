\title{Active Context Compression: Enabling ``Immortal'' Agents via Episodic Memory Consolidation}

\author{\IEEEauthorblockN{Nikhil Verma}

\begin{abstract}
Large Language Model (LLM) agents struggle with long-horizon software engineering tasks due to ``Context Bloat.'' As interaction history grows, computational costs explode, latency increases, and reasoning capabilities degrade due to distraction by irrelevant past errors. Existing solutions often rely on passive, external summarization mechanisms that the agent cannot control. This paper proposes \textbf{Focus}, an agent-centric architecture inspired by the biological exploration strategies of \textit{Physarum polycephalum} (slime mold). The Focus Agent autonomously decides when to consolidate key learnings into a persistent ``Knowledge'' block and actively withdraws (prunes) the raw interaction history. Evaluated on the SWE-bench Lite benchmark using the official Docker-based harness, Focus achieves \textbf{32\% token savings} compared to baseline agents. This compression comes with a trade-off: a modest accuracy reduction (60\% baseline success vs. 40\% Focus success on N=5 instances) due to information loss during aggressive pruning. We demonstrate that active, model-controlled context management provides a valuable engineering trade-off for long-horizon tasks, converting the hard limit of context windows into a sustainable ``cognitive tax.''
\end{abstract}

\begin{IEEEkeywords}
LLM Agents, Software Engineering, Context Management, Memory Consolidation, Autonomous Agents

\section{Introduction}
The ``Context Window'' remains the fundamental bottleneck for autonomous AI agents. While recent models offer massive context capacities (e.g., 200k+ tokens), utilizing this capacity naively presents three critical challenges:
\begin{enumerate}
    \item \textbf{Cost:} In an iterative loop, re-processing a growing history for every new inference step leads to quadratic cost accumulation.
    \item \textbf{Latency:} Time-to-first-token degrades linearly with context length, making interactive agents sluggish.
    \item \textbf{Context Poisoning:} Long contexts filled with trial-and-error logs, failed tests, and verbose tool outputs can distract the model, leading to the ``Lost in the Middle'' phenomenon~\cite{liu2023lost}.
\end{enumerate}

\section{Related Work}

\textbf{Memory-Augmented Agents.} Several approaches address context limitations through external memory. MemGPT~\cite{packer2023memgpt} implements a virtual memory hierarchy inspired by operating systems, allowing LLMs to manage their own memory through explicit read/write operations. Voyager~\cite{wang2023voyager} uses a skill library to store reusable code snippets, reducing redundant exploration.

\textbf{Reflection and Self-Improvement.} Reflexion~\cite{shinn2023reflexion} enables agents to learn from failures by generating textual reflections stored in an episodic memory buffer. However, this operates \textit{between} task attempts rather than within a single continuous trajectory. LATS~\cite{zhou2023lats} combines tree search with LLM reasoning but still accumulates full trajectories.

\textbf{Context Compression.} StreamingLLM~\cite{xiao2023streamingllm} addresses infinite context through attention sink tokens, but operates at the attention level rather than providing agent-level memory management. LLMLingua~\cite{jiang2023llmlingua} compresses prompts but requires a separate compression model.

\textbf{Our Contribution.} Unlike prior work, Focus enables \textit{intra-trajectory} compression---the agent actively prunes its own history \textit{during} a single task, preserving learnings in a structured knowledge block. This complements rather than replaces external memory approaches.

Standard agents operate in an ``Append-Only'' mode: every thought, tool call, and system response is permanently added to the conversation history. This strategy works for short tasks but ensures failure for complex, open-ended exploration, as the agent inevitably hits the token limit or becomes confused by its own previous failures.

We propose a shift from \textit{Passive Retention} to \textit{Active Compression}. Inspired by the efficiency of \textit{Physarum polycephalum}---a slime mold that explores environments and physically retracts from dead ends while leaving chemical markers to avoid re-exploration---we introduce the \textbf{Focus Agent}. This agent is equipped with the ability to introspect, summarize its recent trajectory into a high-level learning, and physically delete the raw logs of that trajectory from its immediate context.

\section{Methodology}

\subsection{The Slime Mold Analogy}
Biological systems do not retain a perfect record of every muscle movement used to navigate a maze; they retain the \textit{learned map}. Similarly, an agent exploring a codebase does not need to remember the 50 lines of \texttt{ls -R} output from ten minutes ago; it only needs to remember that ``the config file is not in the \texttt{/src} directory.''

\subsection{Architecture: The Focus Loop}
The Focus architecture introduces two primitives to the standard ReAct agent loop: \texttt{start\_focus} and \texttt{complete\_focus}. Crucially, the agent has \textit{full autonomy} over when to invoke these tools---there are no external timers or heuristics forcing compression.

\begin{enumerate}
    \item \textbf{Start Focus:} The agent declares what it's investigating (e.g., ``Debug the database connection''). This marks a checkpoint in the conversation history.
    \item \textbf{Explore:} The agent uses standard tools (read, edit, run) to perform work.
    \item \textbf{Consolidate:} When the agent naturally completes the sub-task or hits a dead end, it \textit{decides} to call \texttt{complete\_focus}, generating a summary of:
    \begin{itemize}
        \item What was attempted?
        \item What was learned (facts, file paths, bugs)?
        \item What is the outcome?
    \end{itemize}
    \item \textbf{Withdraw:} The system takes this summary, appends it to a persistent ``Knowledge'' block at the top of the context, and \textit{deletes} all messages between the checkpoint and the current step.
\end{enumerate}

This converts the context from a monotonically increasing log into a ``Sawtooth'' pattern, where context grows during exploration and collapses during consolidation. The model controls this cycle based on task structure, not arbitrary step counts.

\section{Experiments}
We evaluated the Focus architecture on \textbf{SWE-bench Lite}, a benchmark for software engineering agents solving real GitHub issues. All evaluations used the official SWE-bench Docker-based harness to ensure reproducible, leaderboard-quality pass/fail determination. We ran a controlled A/B comparison using \texttt{claude-sonnet-4-5-20250929} on N=5 instances, running both Baseline and Focus agents on identical tasks to measure the efficiency-accuracy trade-off.

\subsection{Key Findings}

\textbf{Finding 1: Significant Token Savings.} Focus achieved a \textbf{32\% reduction in token consumption} (Table~\ref{tab1}), demonstrating that model-controlled compression successfully reduces computational costs. The agent compressed on average 2.6 times per task, dropping 115 messages while preserving key learnings in a structured knowledge block.

\textbf{Finding 2: The Accuracy Trade-off.} Focus achieved 40\% success rate compared to Baseline's 60\% (2/5 vs 3/5 resolved instances). Analysis reveals that Focus failed on \texttt{scikit-learn\_\_scikit-learn-10297}, which Baseline solved. This suggests information loss during compression---when the agent summarizes aggressively, critical details may be discarded. This represents an honest engineering trade-off: substantial cost savings in exchange for a modest accuracy penalty.

\textbf{Finding 3: Context Reduction.} Focus maintained an average of 36.6 messages compared to Baseline's 134.2 messages (73\% reduction). This dramatic reduction in context size demonstrates that the compression mechanism effectively prunes verbose exploration history while retaining task-relevant information.

\subsection{Case Study: The Sawtooth Pattern}
The efficiency gain stems from the ``Sawtooth'' context pattern. The Baseline agent's context grew monotonically to 134 messages per task on average. In contrast, the Focus agent compressed its history 2.6 times per task, dropping raw logs while retaining high-level knowledge, finishing with lean contexts of 37 messages on average. This pattern demonstrates that agents can explore deeply while maintaining manageable cognitive load.

\subsection{Case Study: When Compression Fails}
The Focus agent failed on \texttt{scikit-learn\_\_scikit-learn-10297} despite Baseline succeeding. Analysis of the trajectory reveals that Focus compressed 3 times and dropped 119 messages. It is likely that during one of these compressions, a critical detail about the test expectations was summarized too broadly, causing the agent to implement a partial fix that failed the official test suite. This illustrates the fundamental trade-off: aggressive compression saves tokens but risks information loss.

\subsection{Analysis}
\textbf{Efficiency vs. Accuracy Trade-off:} The Focus Agent reduced token consumption by 32\% but achieved 20\% lower success rate (2/5 vs 3/5). This represents a real engineering trade-off: for long-horizon tasks where cost is a primary concern, Focus offers substantial savings. For tasks where accuracy is paramount, the baseline append-only strategy may be preferable.

\textbf{Model Autonomy Works:} The agent compressed 2.6 times per task on average \textit{without any external prompting}. This demonstrates that capable models can self-regulate their context when given the appropriate tools, validating the core thesis of agent-controlled compression.

\textbf{Context Reduction:} By maintaining a context size of 37 messages on average (vs. 134 for baseline), Focus mitigates the quadratic cost of re-processing long histories while maintaining sufficient information to complete most tasks successfully.



\section{Discussion}
\subsection{The Cost of Immortality}
Active compression introduces a ``cognitive tax''---the tokens spent generating summaries. We measured this tax: in our experiments, Focus agents generated on average 11,556 output tokens (for Haiku) compared to 18,158 for baseline agents. The \textit{additional} tokens for summary generation are offset many times over by the input token savings from not re-processing history.

For short, trivial tasks, this tax may outweigh the benefits. However, for long-horizon exploration (40+ steps), the tax becomes negligible compared to the cost of carrying a massive history. It effectively amortizes the cost of ``memory'' over the lifetime of the agent.

\subsection{Limitations}
Several limitations warrant discussion:

\begin{enumerate}
    \item \textbf{Sample Size:} Our evaluation uses N=5 instances. While sufficient to demonstrate the efficiency-accuracy trade-off, the high computational cost of evaluating long-horizon agentic workflows using the official Docker harness limited our sample size. How these results generalize to the full benchmark (N=300) remains to be seen.
    
    \item \textbf{Information Loss:} Compression necessarily discards details. We observed one clear case (\texttt{scikit-learn\_\_scikit-learn-10297}) where aggressive compression led to task failure. The challenge is balancing token savings against the risk of losing critical information. Future work could explore more sophisticated compression strategies that preserve key details more reliably.
    
    \item \textbf{Model Dependency:} Our results are based on Claude Sonnet 4.5. How other models (GPT-4, Gemini, etc.) perform with Focus remains an open question. Different models may have varying abilities to self-regulate compression effectively.
\end{enumerate}

\section{Conclusion}
We have demonstrated that model-controlled active context management provides a valuable engineering trade-off for long-horizon LLM agent tasks:

\begin{enumerate}
    \item \textbf{Substantial Cost Savings:} Focus reduces token consumption by 32\% and message count by 73\%, translating directly to reduced API costs and faster execution.
    
    \item \textbf{Modest Accuracy Cost:} This compression comes with a 20\% reduction in success rate, representing an honest trade-off between efficiency and correctness.
\end{enumerate}

The key insight is that LLMs, when given appropriate tools, can autonomously manage their own context by deciding when to compress completed work. This converts the hard limit of context windows into a manageable ``cognitive tax'' that enables deeper exploration on long-horizon tasks.

This finding has practical implications: for cost-sensitive applications where occasional failures are acceptable (exploratory coding, prototyping, research), Focus offers substantial savings. For mission-critical applications requiring maximum accuracy, the baseline approach remains preferable.

The Focus architecture provides a practical, complementary approach to existing memory management techniques, operating entirely within the context window without requiring external infrastructure. It demonstrates that agents can be given agency over their own memory management, opening the door to more sophisticated self-regulating systems.

Future work will explore more sophisticated compression strategies that better preserve critical information, hierarchical knowledge structures, and the relationship between model capability and compression effectiveness.
